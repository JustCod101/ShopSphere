## 三、系统在背后做了什么(性能与稳定性)

商品页是电商**流量最大**的页面(经验上占总访问的八九成)。这一阶段的重头戏不是"能不能显示",
而是"**被海量访问时扛不扛得住**"。做了三件事:

### ① 商品详情页为什么能扛住高并发 —— 缓存

**类比**:每次查商品都去数据库,等于每个顾客问价都让店员跑一趟仓库档案室 —— 人一多就堵死。
我们在"前台"放了一张**速查表**(Redis 缓存):第一个人问过之后,信息就贴在速查表上,
后面的人秒级拿到,不再跑仓库。

同时防住了三种典型事故(都是行业内的经典坑):

| 事故 | 通俗解释 | 我们的对策 |
|---|---|---|
| **缓存穿透** | 有人反复查一个根本不存在的商品,每次都把请求捅到数据库 | 连"不存在"也记在速查表上,挡住反复捅 |
| **缓存击穿** | 某个爆款的速查记录刚好过期,瞬间一大批请求同时涌向数据库 | 加锁,只放一个请求去查,其余等它回填 |
| **缓存雪崩** | 大批速查记录在同一秒集体过期,数据库被瞬间打爆 | 给每条记录的过期时间加随机抖动,错峰失效 |

还加了**命中率监控**:运维能实时看到"速查表帮我们挡掉了多少流量"。

### ② 怎么保证不超卖 —— 库存原子预扣

**类比**:库存就像医院的**叫号机**。100 个号,发完即止;无论多少人同时点,
也绝不会出现两个人拿到同一个号。

我们用 Redis + 一段保证"不可被打断"的脚本实现:**扣库存这个动作要么整体成功、要么整体不发生**,
中间不会被其他请求插队。已做**并发压力验证**:50 个线程同时抢 100 件库存、共发起 1000 次抢购,
结果**正好 100 次成功、其余全部被正确拒绝、最终库存归零** —— 既不超卖,也不少卖。

另外加了一个**对账巡检**:每 5 分钟自动核对"快速库存(Redis)"与"账本库存(数据库)"是否一致,
对不上就报警,人工介入 —— 库存涉及钱和信誉,我们选择"发现即报警",不做无人值守的自动改数。

### ③ 为"下单"打的地基 —— 库存预留协议

下单是个**多步骤**动作,不能一步到位。我们采用电商通用的"三段式"思路:

```
下单时：先「预留」库存（Try）—— 货还在，但别人抢不到了
付款成功：「确认」扣减（Confirm）—— 真正出库
超时没付：「取消」释放（Cancel）—— 把货放回货架
```

这一阶段把这三段的**接口骨架**和**订单服务↔商品服务之间的对接协议**都定好了。
打个比方:两个部门要协作,先把"对接表单的格式"敲定 —— 这样下一阶段做订单时,
可以直接照着协议对接,不用返工。

> 说明:三段式的**完整严密版本**(异常顺序、重复请求的极端情况)留到下一阶段(T3.3)和
> 订单服务一起打磨。本阶段交付的是**协议 + 骨架**,够下一步开工。

## 五、工程质量保障

- **自动化测试**:商品服务累计 **75 个测试**,覆盖查询 / 缓存命中与失效 / 库存并发 / 接口骨架等,
  每次改动全跑通过才提交。
- **真实环境验证库存**:库存测试不用"模拟",而是**拉起一个真实的 Redis** 来跑 —— 更接近线上,
  并发安全性是实测出来的,不是推测的。
- **缓存效果可量化**:配套压测脚本 + 命中率埋点,缓存带来的提速可以拿出真实数字。
- **契约文档同步**:接口规范、缓存设计、库存设计都有对应文档(`docs/`),改代码必改文档。
- **提交可追溯**:每次提交说明写清"做了什么、为什么、风险在哪"。

## 七、潜在风险与已采取的对冲

| 风险 | 影响 | 当前对冲 |
|---|---|---|
| Redis 故障 | 商品详情 / 库存读取受影响 | 详情可回退查数据库;生产需对 Redis 做高可用(哨兵/集群) |
| 库存"快速账"与"数据库账"漂移 | 可能少卖或超卖 | 每 5 分钟自动对账 + 报警,人工介入;不做无人值守自动改数 |
| 下单三段式骨架未严密 | 极端并发下可能不一致 | 本阶段骨架未对接任何调用方,不影响线上;严密版随 T3.3 落地 |
| 库存测试依赖 Docker | 无 Docker 的机器跑不了该测试 | 已在构建里固化兼容配置;CI 需提供 Docker 环境 |

## 八、对业务方的建议

1. **商品演示数据**:目前内置 20 个商品是占位数据,正式上线前需导入真实商品 —— 建议尽早确定
   商品字段是否够用(现有:名称/类目/价格/图片/描述/库存)。
2. **库存初始值**:演示数据每个商品库存 100,真实库存由后续后台/导入确定。
3. **缓存过期时间**:商品详情默认缓存 30 分钟。如果业务上商品信息改动后要求"立刻"生效,
   需走后台编辑接口(后续开放),它会主动刷新缓存。
4. **支付超时窗口**:下单后多久不付款就释放库存(行业常见 30 分钟),这个数字需要业务方拍板 ——
   会影响 T3.3 的实现。
