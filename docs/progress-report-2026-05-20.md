## 一、项目是做什么的

ShopSphere 是我们正在搭建的**微服务电商平台底座**。"微服务"可以理解为:整个系统拆成多个独立的小服务,各管一摊 —— 一个负责用户、一个负责商品、一个负责订单……这样团队可以并行开发,某一块出问题不会拖垮全站,扩容也可以只扩压力大的部分。

到今天为止,我们完成了**整套地基与第一个完整业务服务(用户服务)**,并已经具备应对真实流量的基础防护能力。

## 二、做了哪些事(按用户价值分组)

### ① 用户能干什么(对外能力)

| 能力 | 状态 | 说明 |
|---|---|---|
| **注册账号** | ✅ 已上线 | 用户名 + 密码 + 邮箱/手机,密码用业界标准强加密(BCrypt)存储,数据库即便泄漏也无法反推原密码 |
| **登录拿到令牌** | ✅ 已上线 | 登录成功签发一个有效期 2 小时的"通行证"(JWT),后续所有请求带这个令牌,无需重复登录 |
| **查自己的资料** | ✅ 已上线 | 携带令牌即可查询当前账号信息 |
| **记录浏览/加购/下单行为** | ✅ 已上线 | 用户在前端的操作可上报到服务端,后续用于推荐系统训练 |
| **服务之间内部查用户** | ✅ 已上线 | 其他服务(如未来订单服务)需要查用户信息时,走专属内部接口,不暴露给公网 |

### ② 系统怎么保护自己(安全 / 稳定性)

| 防护 | 状态 | 通俗解释 |
|---|---|---|
| **统一入口(网关)** | ✅ | 所有外部请求只能通过网关进来,内部接口直接挡掉。类似商场只有一个正门,后勤通道不对顾客开放 |
| **令牌真伪校验集中在网关** | ✅ | 进门时验一次令牌,后续服务都信任,不重复检查。简化业务代码、统一安全策略 |
| **登录失败防爆破** | ✅ | 同一账号 10 分钟内错 5 次密码,自动锁 30 分钟。类似银行卡输错三次锁卡 |
| **限流防洪峰(Sentinel)** | ✅ | 给每个敏感接口设流量上限:登录 20 次/秒、注册 5 次/秒、行为上报 200 次/秒。超过就拒绝,保护后端不被打挂。两层防御:网关挡一道,业务服务再兜一道 |
| **链路追踪 ID** | ✅ | 每个请求带一个唯一标识贯穿全链路,出问题时能精确还原一次请求经过了哪些服务、在哪一步失败 |
| **令牌密钥可零停机轮换** | ✅ | 签名密钥泄漏或定期轮换时,无需重启任何服务 |
| **配置中心热更新** | ✅ | 限流阈值、白名单、超时时间……运营人员在配置中心改了,服务几秒内自动生效,不用重新发版 |
| **密钥不入代码仓库** | ✅ | 数据库密码、签名私钥等敏感信息全部加密存放,即便代码外泄也拿不到生产凭据 |

### ③ 数据怎么流(业务管道)

我们设计并实现了一条**事件驱动**的数据通路,为后续推荐系统铺好路:

```
用户在网页浏览/加购/下单
        ↓
       网关
        ↓
  用户服务(写入自己的数据库做审计)
        ↓
  消息总线(RabbitMQ)
        ↓
  推荐系统(Phase 4 接入,有自己独立的数据库)
```

**关键设计**:推荐系统**不直接读用户服务的数据库** —— 这是一条铁律,避免两个团队改库互相踩脚、避免数据库被推荐侧高频查询拖垮。两边通过消息总线解耦,各管各的数据。

## 六、潜在风险与已采取的对冲

| 风险 | 影响 | 当前对冲手段 |
|---|---|---|
| RabbitMQ 短暂故障 | 用户行为事件可能丢几条(不影响下单等关键链路) | 已选择"轻量级"可靠性档位,故障日志可告警,本地数据库仍有自审计副本 |
| Redis 短暂故障 | 防爆破计数暂时失效 | 程序选择"放行"避免业务全挂,但记 ERROR 日志触发告警 |
| Sentinel Dashboard 宕机 | 规则仍正常工作,只是看不到实时监控 | 规则源是配置中心(Nacos),Dashboard 仅观测,不影响业务 |
| Nacos 配置中心宕机 | 客户端会用本地缓存的最后一次规则,服务不会卡死 | 已在设计中确认 fail-safe 行为,生产环境需对 Nacos 单独做高可用 |
| 单点性能压力 | 单机限流够 demo,但生产高并发需要集群版限流 | 已在文档中注明,Phase 2 视实际流量评估升级 |

## 七、对业务方的几个具体建议

1. **限流阈值需要业务方拍板**:目前用的数字(登录 20 QPS、注册 5 QPS、行为 200 QPS)是技术初值,**真实运营压力下可能需要调整**。建议正式上线前根据预期注册转化率做一次压测调优 —— 我们的脚本随时可跑。
2. **行为埋点字段定义**:目前接受 `view / cart / order` 三种行为类型 + 可选扩展字段。如果产品要追踪更细的事件(如"加入收藏""分享商品"),请尽早提需求 —— 改动小,但越早确定越好。
3. **运营后台权限规划**:配置中心(Nacos)目前所有开发都能改限流参数,生产环境建议加权限审批流程 —— 这是 Phase 5 运维治理的事项。
4. **令牌有效期 2 小时**:目前是技术默认值。如果业务上希望"用户长时间不操作就重登"或"7 天免登录",请明确 —— 改一个配置数字即可,但需要业务侧拍板。
